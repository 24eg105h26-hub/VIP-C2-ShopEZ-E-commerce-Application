1.1 PROJECT VISION & GOALS

ShopEZ is a production-ready B2C e-commerce marketplace built on the MERN stack (MongoDB, Express.js, React.js, Node.js). The platform provides fully isolated portals for Customers, Sellers, and Administrators with enterprise-grade security, real-time notifications, and multiple payment gateway integration.

PROJECT NAME: ShopEZ PROJECT TYPE: B2C E-Commerce Marketplace TECHNOLOGY: MERN Stack (MongoDB, Express.js, React.js, Node.js) TARGET LAUNCH: Week 16 (December 2026) TEAM SIZE: 10 members

CORE OBJECTIVES:

1. Functionality: Deliver complete CRUD operations for all entities with zero data loss 2. Performance: Achieve page load times under 2 seconds and search response under 500ms 3. Security: Implement military-grade authentication and authorization with JWT refresh token rotation 4. Scalability: Support 10,000+ concurrent users with horizontal scaling capability 5. User Experience: Provide intuitive, responsive interfaces across all portals 6. Reliability: Maintain 99.9% uptime with proper error handling and monitoring

PROJECT SCOPE

In Scope: • Multi-role authentication system (Customer, Seller, Admin) • Complete product catalog with variant management • Shopping cart and secure checkout process • Multiple payment gateways (Stripe and Razorpay) • Complete order management system with status tracking • Seller dashboard with inventory and analytics • Comprehensive admin dashboard • Product reviews and rating system • Wishlist functionality • Real-time WebSocket notifications • Full-text search with advanced filtering • Email notification system • Role-based access control (RBAC)

Out of Scope: • Mobile native applications (iOS/Android) • AI-based product recommendations • Subscription-based products • Live chat support system • Advanced inventory forecasting • Multi-language support • Advanced CMS features

1.2 PROBLEM ANALYSIS

Market Challenge: Small sellers and businesses struggle to find affordable, easy-to-use online marketplace solutions that provide both features and security, while customers seek trustworthy platforms with diverse products and secure transactions.

IDENTIFIED PROBLEMS:

Problem 1: Fragmented E-Commerce Solutions Current marketplace platforms lack clear separation between customer, seller, and admin responsibilities, leading to confusion and operational inefficiency. Existing solutions combine these roles, creating security risks and complicated user experiences.

Problem 2: Security Vulnerabilities Most e-commerce platforms suffer from weak authentication mechanisms, insufficient rate limiting, missing CSRF protection, and inadequate security headers. These vulnerabilities expose user data and payment information to risks.

Problem 3: Inventory Management Gaps Traditional inventory systems lack variant-level tracking for product specifications (size, color, dimensions). This leads to overselling, customer dissatisfaction, and complex manual reconciliation processes.

Problem 4: Real-Time Communication Deficiency Current solutions lack real-time order status updates and inventory notifications, forcing customers and sellers to manually refresh pages or check emails repeatedly.

IMPACT ASSESSMENT:

Severity: High (affects core business operations) Frequency: Continuous (millions of daily transactions globally) User Base Affected: Billions of online shoppers and sellers worldwide Current Market Gap: Large and growing (enterprise solutions too expensive)

1.3 PROPOSED SOLUTIONS

Solution 1: Three-Portal Architecture Implement completely isolated portals for Customers, Sellers, and Administrators: • Customer Portal: Browse, search, purchase, track orders, review products • Seller Portal: Manage inventory, view sales analytics, fulfill orders • Admin Portal: System oversight, user management, dispute resolution

Solution 2: Advanced Security-First Design • JWT authentication with refresh token rotation (RTR) • Bcryptjs password hashing (12 salt rounds) • Rate limiting on all endpoints (5-100 requests/minute) • Input validation and sanitization via express-validator • Security headers via Helmet.js • HTTPS enforcement with secure, HttpOnly cookies

Solution 3: Variant-Based Inventory Management • Product variants (size, color, specifications) with individual SKUs • Real-time inventory locking during checkout • Low stock alerts and inventory warnings • Bulk inventory management for sellers • Automatic stock adjustment on order processing

Solution 4: Real-Time Notification System • WebSocket implementation via Socket.io for instant updates • Namespace isolation for customer/seller/admin events • Email notifications via Nodemailer for key events • In-app notification badges with sound alerts • Notification preferences management

Solution 5: Multiple Payment Processing • Stripe integration for global payments • Razorpay integration for India-specific payments • Webhook signature verification for security • Idempotency key handling to prevent duplicates • Automatic refund processing • Payment status tracking and reconciliation

2.1 FUNCTIONAL REQUIREMENTS OVERVIEW

AUTHENTICATION & AUTHORIZATION (FR-1)

FR-1.1 User Registration Actors: Anonymous Users Description: Users can register as Customer or Seller with email verification

Key Features: • Email and password validation (min 8 chars, special characters required) • Unique email enforcement • Verification email sent within 5 seconds • Registration token valid for 24 hours • Role selection (Customer/Seller) • Account creation without instant activation

Acceptance Criteria: ✓ Password must be 8+ characters with uppercase, lowercase, number, special char ✓ Email format validation with DNS check ✓ Unique email enforcement in database ✓ Verification email sent and logged ✓ User can register in < 3 seconds

FR-1.2 Email Verification Actors: Registered Users Description: Verify user email ownership before enabling full access

Key Features: • One-time verification link (valid 24 hours) • Secure token generation and validation • Automatic account activation on successful verification • Email re-send functionality

Acceptance Criteria: ✓ Token generates unique, secure value ✓ Token valid for exactly 24 hours ✓ Token is single-use (invalidates after first use) ✓ Clear error messages for expired/invalid tokens ✓ Re-send available after 5 minutes

FR-1.3 User Login Actors: Registered, Verified Users Description: Authenticate users with email and password

Key Features: • Credential validation against hashed passwords • JWT access token generation (15-minute validity) • Refresh token generation (7-day validity) • Secure HTTP-only cookie storage • Failed attempt logging and rate limiting • Concurrent session management

Acceptance Criteria: ✓ Login process completes in < 500ms ✓ Secure cookie flags set (HttpOnly, Secure, SameSite) ✓ Failed attempts logged for security audit ✓ Rate limited to 5 attempts per minute per IP ✓ Account lockout after 10 failed attempts (15 min lockout) ✓ Last login timestamp updated

FR-1.4 Refresh Token Rotation (RTR) Actors: Authenticated Users Description: Maintain secure sessions with automatic token refresh

Key Features: • Automatic refresh when access token expires • New access and refresh token generation • Old refresh token invalidation (reuse detection) • Secure token storage in database • Token blacklisting for revoked tokens

Acceptance Criteria: ✓ Refresh process completes in < 100ms ✓ Detects and prevents token reuse attacks ✓ Invalid refresh token returns 401 Unauthorized ✓ New token pair issued simultaneously ✓ Old token immediately blacklisted

FR-1.5 User Logout Actors: Authenticated Users Description: Terminate user session and invalidate tokens

Key Features: • Refresh token invalidation • Session cookie clearing • Logout event logging • Immediate effect across all devices

Acceptance Criteria: ✓ Token invalidation effective immediately ✓ Reusing old token returns 401 error ✓ Logout event logged with timestamp ✓ User redirect to login page

FR-1.6 Account Deletion Actors: Authenticated Users Description: Allow users to permanently delete their account

Key Features: • Password re-verification requirement • Data anonymization (not permanent deletion for audit trail) • Personal information removal • Transaction history preservation (anonymized)

Acceptance Criteria: ✓ Requires password confirmation ✓ Personal data deleted immediately ✓ Account marked as deleted (soft delete) ✓ Transaction records anonymized ✓ Confirmation email sent to registered email

ARCHITECTURE PATTERN

Clean Architecture with Separation of Concerns:

Presentation Layer (Frontend) ↓ (HTTPS/WebSocket) API Layer (Express.js) ├── Routes (endpoint definitions) ├── Controllers (request handling) ├── Middleware (auth, validation, errors) └── Services (business logic) ↓ Data Access Layer └── Mongoose Models (database queries) ↓ Database Layer (MongoDB)

Benefits: • Testability: Each layer can be tested independently • Maintainability: Clear separation of concerns • Scalability: Services can be extracted to microservices • Reusability: Services shared across multiple routes

4.4 SECURITY ARCHITECTURE

Authentication Flow:

1. User Registration: ├── Collect email, password, name, role ├── Validate input (express-validator) ├── Hash password with bcryptjs (12 salt rounds) ├── Generate verification token ├── Save user to database └── Send verification email

2. Email Verification: ├── User clicks link in email ├── Extract and validate token ├── Mark user as verified ├── Invalidate token └── Enable login access

3. User Login: ├── Collect email and password ├── Query user from database ├── Compare password with hash (bcryptjs) ├── Generate JWT access token (15 min validity) ├── Generate refresh token (7 day validity) ├── Store refresh token in database (hashed) ├── Set HttpOnly, Secure cookies └── Return tokens to client

4. Request Authentication: ├── Client sends request with access token (cookie) ├── Middleware extracts token from cookie ├── Verify JWT signature ├── Extract user ID from payload ├── Check token expiration ├── Allow or deny request └── Proceed to controller/service

5. Refresh Token Rotation: ├── Client sends refresh token (cookie) ├── Extract token from cookie ├── Query database for stored token ├── Verify token hash matches ├── Check token hasn't been used before (reuse detection) ├── Generate new access token ├── Generate new refresh token ├── Invalidate old refresh token ├── Store new refresh token in database (hashed) ├── Set new cookies └── Return new tokens

---

SECURITY MEASURES

Rate Limiting: • Auth endpoints: 5 requests per minute per IP (prevents brute force) • API endpoints: 100 requests per minute per user (prevents abuse) • Search: 50 requests per minute per IP (prevents DOS) • Uploads: 10 requests per minute per user (prevents spam)

Input Validation: • Email: RFC 5322 format validation + DNS verification • Password: Min 8 chars, uppercase, lowercase, number, special char • Phone: E.164 format validation • URLs: URL format validation • Custom regex for specific fields • Sanitization of SQL/XSS injection attempts

HTTPS & Secure Cookies: • All communication encrypted with TLS 1.3 • Cookies: HttpOnly (prevents JavaScript access) • Cookies: Secure (HTTPS only) • Cookies: SameSite=Strict (CSRF protection) • Certificate pinning (Phase 2)

Payment Security: • No card data stored (PCI DSS Level 1 compliance via Stripe) • Webhook signature verification • Idempotency keys for payment requests • Rate limiting on payment endpoints • Encryption of sensitive fields (bank details for sellers)

Data Encryption: • Passwords: Bcrypt with 12 salt rounds • Sensitive fields: AES-256-GCM encryption at rest • Database: MongoDB encryption at rest • Backups: Encrypted snapshots • API communication: TLS 1.3

Access Control (RBAC): • Customer: View own data, create orders, write reviews • Seller: View own products/orders, manage inventory • Admin: Full platform access, dispute resolution • Middleware: Role verification on protected endpoints

Deployment Steps:

Stage 1: Infrastructure Preparation 1. Verify production environment status 2. Create database backup 3. Verify SSL certificates 4. Check monitoring and alerting 5. Prepare rollback plan

Stage 2: Staged Rollout 1. Deploy to 10% of users (canary deployment) - Monitor error rates, response times - Collect performance metrics - Wait 30 minutes, verify stability

2. Deploy to 50% of users (progressive rollout) - Monitor key metrics - Verify feature functionality - Collect user feedback - Wait 1 hour, verify stability

3. Deploy to 100% of users (full rollout) - Monitor closely for 24 hours - Address any critical issues - Schedule post-mortem if issues found

Stage 3: Post-Deployment 1. Verify all systems operational 2. Test all critical user journeys 3. Check analytics and error logs 4. Update documentation 5. Communicate with stakeholders 6. Schedule team retrospective

Rollback Procedure: If critical issues detected: 1. Immediately halt deployment to remaining users 2. Revert to previous stable version 3. Restore from database backup if needed 4. Verify all systems operational 5. Notify stakeholders and customers 6. Document incident and root cause 7. Schedule incident postmortem

Rollback steps: • Git: git revert • Database: Restore from backup (automated, < 5 minutes) • Frontend: Clear CDN cache, revert to previous build • Services: Restart with previous configuration • Monitoring: Validate system health
